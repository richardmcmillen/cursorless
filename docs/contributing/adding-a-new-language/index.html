Adding a new programming language

So you want cursorless to support a new language? Great! Here's how to do it:

1. Add support to the vscode-parse-tree extension‚Äã

Cursorless relies on the vscode-parse-tree extension to access the parse tree of a document. See the docs there for how to add support for a new parser

2. Define parse tree patterns in Cursorless‚Äã

First a few notes / tips:

  • We suggest opening a draft PR as soon as possible to get early feedback. Please use the new language PR template either by adding ?template=new_programming_language to the end of the URL you used to open the PR, or just by copying and pasting from the template to your PR body, if that's easier.
  • We suggest adding tests as early as possible, after each language feature you add. Recording tests is quick and painless using the test case recorder described below. We promise üòá

Minimum changes that each language needs:

  • new file in /src/languages/<yourlanguage>.ts. Take a look at existing languages as a base. At its core you're implementing your language's version of the nodeMatchers const, mapping scope types found in Types.ts:ScopeType with matching expressions that align with the parse tree output.
  • new entry in getNodeMatcher.ts:languageMatchers, importing your new file above
  • new entry in constants.ts
  • new text fragment extractor (default is likely fine) in getTextFragmentExtractor.ts:textFragmentExtractors

The parse trees exposed by tree-sitter are often pretty close to what we're looking for, but we often need to look for specific patterns within the parse tree to get the scopes that the user expects. Fortunately, we have a domain-specific language that makes these definitions fairly compact.

  • Check out the docs for the syntax tree pattern matcher
  • If you look in the debug console, you'll see debug output every time you move your cursor, which might be helpful.
  • You will likely want to look at node-types.json for your language, (eg java). This file is generated from grammar.js, which might also be helpful to look at (eg java).

Writing tests‚Äã

Test cases can be automatically recorded, which should speed things up a lot. See the docs for the test case recorder. It will also likely be helpful to look at the existing recorded test cases (eg java) to see how they should end up looking when they're recorded.
